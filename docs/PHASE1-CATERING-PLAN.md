## 1. 客户一期目标

```
前端 - 小程序（一期 餐饮）— 独立 Taro 工程，通过 API 访问本项目后端
  1. 素材：店铺信息、菜品信息
  2. 海报：模版海报、原生海报
  3. 营销：发布管理、统计信息

后台 - Web-PC（本项目 Blazor 保留）
  - 商户管理
  - 海报模版
```

**本项目职责**：
- 后端业务内核（模板、会话、素材、发布、统计、AI 编辑、渲染）
- **完整的 Web API 层**（供独立 Taro 小程序调用；必须完全 API 化，不留 Blazor 直调 Service 的暗接口）
- PC 运营后台（Blazor，商户管理 + 模板管理）

**独立 Taro 工程**（不在本项目仓库，但**是一期整体交付的一部分**）：
- 微信小程序前端（Taro 3 + React）
- 通过 HTTPS + Bearer token 调用本项目 API
- 由前端团队/另一个仓库负责实现，本项目提供并维护 API 契约

## 2. 参考图分析：客户想要的海报长什么样

客户提供 6 张参考海报（居酒屋菜单、六福家菜、丹姐大排档、意面店、大席小炒、土川菜），共同特征：

| 元素 | 表现 | 现有能力 |
|------|------|----------|
| 多分区布局（烤/炸/主食/凉菜…） | 每张 3~8 个分区，每区标题+菜品 | ⚠️ 缺**分组容器**节点 |
| 菜品网格（图+名+原价+促销价） | 每区 3~10 个菜品重复排布 | ✅ 已有 `repeat`（多列） |
| 主打菜大图 + 周边小图 | 图 3/6 中心大图+周边小菜 | ✅ 组合支持 |
| 折扣价红底高亮 | 原价+划线+促销价红底 | ✅ `price` + `rect` |
| 品牌 LOGO + 店铺名 | 固定角落 | ✅ image + text |
| 二维码/地址/电话 | 角落 | ✅ 已支持 |
| **多规格画布**（横/竖/方/长图/A4） | 6 张参考图无一相同，比例从 1:3 到 3:2 都有 | ⚠️ **渲染器已支持**、种子模板全竖版、预览 CSS 写死了竖版 |
| 背景装饰（图案/整体色调） | 品牌视觉 | ⚠️ `canvas.background` 仅纯色 |
| 分类标题装饰（胶囊/艺术字） | "精品菜品" 徽章 | ⚠️ 缺**胶囊/徽章预设** |

**结论**：模板协议核心（文字/价格/图片/二维码/重复）都够用，参考图这种"菜单式海报"需要**5 处增量扩展**（**画布规格白名单**、分组容器、网格 repeat 校验、背景图、徽章预设），不用推倒重来。

**关于画布尺寸**：`PosterRenderService` 早已把 `canvas.width/height` 参数化（不是硬编码），任何尺寸都能渲染。真正要做的是：
- 定义**常用规格白名单**（下方 §4.3），给运营选择时用
- 修预览 CSS 里写死的 1080:1920 比例，改成读 `canvas.width/height` 动态计算
- 补几套典型比例的种子模板

### 4.3 模板协议扩展（参考图驱动，是本期真正的技术难点）

| 扩展点 | 现状 | 目标 |
|--------|------|------|
| **画布规格** | 渲染器已参数化，种子模板全竖版；预览 CSS 写死 1080:1920 | 定义**常用规格白名单**（见下）；预览 CSS 改为动态读比例；补 3~5 套典型规格的种子模板 |
| 分组容器 | `layout` 拉平 | 新增 `section` 节点：`{type:section, x, y, w, h, children:[...]}` |
| 网格 repeat | 已支持 columns | 补测试用例+菜品网格模板 |
| 背景图 | `canvas.background` 只支持颜色 | 扩展为 `{type:solid\|image\|gradient, ...}` |
| 徽章/胶囊 | 缺预设 | 定义 `badge`/`pill` 复合节点（可用 rect+text 实现，出 3~5 个预设） |

**画布规格白名单**（覆盖参考图和主流投放场景，全部按短边 1080px 基准）：

| 规格 | 尺寸(px) | 比例 | 典型用途 | 参考图对标 |
|------|---------|------|---------|-----------|
| `poster-portrait` | 1080×1920 | 9:16 | 单菜品海报、朋友圈竖图 | 图 3 变体 |
| `poster-square` | 1080×1080 | 1:1 | 朋友圈方图、微信 tabBar 缩略 | — |
| `menu-landscape` | 1920×1080 | 16:9 | 广告机、简易横版 | — |
| `menu-a4-portrait` | 1240×1754 | ≈A4 竖 | 打印菜单、A4 展架 | 图 3 |
| `menu-a4-landscape` | 1754×1240 | ≈A4 横 | 打印菜单横版 | 图 2、图 5 |
| `menu-trifold` | 2480×1240 | 2:1 | 三折页、长横菜单 | 图 4、图 6 |
| `menu-panorama` | 3600×1240 | ≈3:1 | 超长横菜单/长条展架 | 图 1、图 6 变体 |
| `poster-longform` | 1080×3240 | 1:3 | 详情长图、公众号推文长图 | — |

> 白名单不是硬约束，模板可以自己设 `canvas.width/height` 用任意尺寸；白名单只是给运营新建模板时的推荐选项。

### 4.4 横切基础设施

1. **登录鉴权**：微信 `code2session` + PC 账号登录（当前完全没有）
2. **API 化 + CORS**：小程序只能走 REST/JSON，需 Bearer token + 跨域
3. **持久化补强**：会话消息/版本/素材/发布/统计需入库
4. **文件存储**：MVP 用本地磁盘，二期换 MinIO
5. **PNG 渲染**：现有 `IRenderService` 只输出 HTML；一期最小实现（推荐 Playwright/无头浏览器方案，复用 HTML 渲染管线）

## 5. 一期技术架构

```
┌──────────────────────────────┐        ┌──────────────────────────┐
│  微信小程序（商户端）         │        │  Web-PC 运营后台         │
│  Taro 3 + React              │        │  Blazor Server           │
│  【独立工程 / 另一仓库】     │        │  【本项目内】            │
│  - tabBar：素材/海报/营销/我 │        │  - 商户管理              │
│  - 全部页面原生渲染          │        │  - 模板管理/审核/授权    │
│  - HTTP 调后端 API            │        │                          │
└──────────────┬───────────────┘        └────────────┬─────────────┘
               │  https + Bearer(JSON)               │
               │  ← API 契约由本项目定义并维护       │
               ▼                                     ▼
        ┌───────────────────────────────────────────────────┐
        │  ASP.NET Core 8（本项目）                         │
        │  Web 层：Web API 控制器 + Blazor（PC 后台）       │
        │  Application：模板/会话/素材(新)/发布(新)/统计(新)│
        │  Infrastructure：EF Core + 文件存储 + LLM 调用    │
        │  Worker：PNG 导出、异步 AI 任务                   │
        └───────────────────────────┬───────────────────────┘
                                    ▼
                       SQLite（一期）+ 本地文件存储
```

**关键约定**：
- **Taro 小程序仓库独立**，但本项目必须保证 API 完整、稳定、有版本，供其调用。
- 完整 API 化意味着：**每个能被小程序用到的功能都必须有对应的 REST API**（包括 §8 全部端点）；不能出现"这个只在 Blazor 里能用"的暗接口。
- 现有 Minimal API 骨架保留、扩展；PC 后台复用现有 Blazor 页面。
- 现有的 `ShopEditor` 对话页在 MVP 阶段**降级为内部测试工具**（PC 后台内可访问），生产入口交给小程序。

## 6. 术语澄清：模版海报 vs 原生海报（MVP 版）

- **模版海报**：运营在后台做的固定版式，商户对话替换内容（店名/菜品/价格/图/二维码），不能改结构。参考图 1/2/3/6 这种"多分区菜单"就是靠这个做出来的。
- **原生海报**（MVP 降级）：先实现为"从少量空白/极简模板中选择起点"，不做真自由画布。二期再考虑 Canvas 自由编辑。

## 8. API 切片（一期新增/改造）

**认证**
- `POST /api/auth/wx-login`（code→openid→token）
- `POST /api/auth/admin-login`

**素材**
- `GET/POST/PUT/DELETE /api/shop/profile`
- `GET/POST/PUT/DELETE /api/shop/dishes`
- `POST /api/assets/upload`（图片上传，替换 dataUrl 临时方案）

**海报**
- `GET /api/shop/templates`（行业+授权+审核状态过滤，已有雏形）
- 对话编辑复用 `POST /api/shop/sessions/*`
- `POST /api/shop/works` / `GET /api/shop/works`（作品保存/列表）
- `GET /api/render/{id}.png`（**MVP 补 PNG 渲染**，见 §10 风险）

**营销**
- `POST /api/shop/works/{id}/publish`
- `GET /api/stats/summary`（门店统计）
- `POST /api/stats/event`（埋点上报）

**后台**
- `GET/POST/PUT /api/admin/shops`（商户管理）
- 模板管理复用 `/api/admin/templates/*`

## 9. 实施计划（MVP 五阶段）

按"先地基、再商户主链路、后营销闭环"排期，每阶段独立可演示。

### M0 · 地基与打通（约 1.5 周）
1. 后端补 Web API 控制器骨架 + CORS + Bearer 鉴权。
2. **PNG 渲染服务从零建**（见 §10 风险 1），预览与导出共用同一管线。
3. 微信小程序注册、企业主体、合法域名 HTTPS。
4. **产出 API 契约文档**（OpenAPI/Swagger），供独立 Taro 工程对接。
- **验收**：
  - `GET /api/render/{sessionKey}.png` 返回正确竖/横版 PNG。
  - Swagger 文档可访问，Taro 团队能拿到完整 API 契约开工。

### M1 · 模板协议扩展 + 参考图落地（约 1.5 周，本期新增）
1. 修预览 CSS 里写死的 1080:1920 比例，改成动态读 `canvas.width/height`。
2. 定义**画布规格白名单**（见 §4.3），后台新建模板时按白名单选。
3. 新增 `section` 分组节点，支持整体位移/背景。
4. `canvas.background` 扩展为对象（solid/image/gradient）。
5. 定义 3~5 个复合节点预设（分区标题胶囊、促销价徽章）。
6. **产出 3 张对标参考图的模板**（1 张 A4 竖菜单 + 1 张 A4 横菜单 + 1 张三折页横菜单）。
- **验收**：三种规格都能正确渲染出 PNG；商户可对话替换内容；预览页对任意画布比例都能自适应。

### M2 · 素材管理（约 1 周）
1. `ShopProfile` + 店铺信息 API。
2. `Dish` + 菜品 CRUD + 图片上传落库（本地存储）。
3. 对话编辑器接入门店素材：选模板后自动带入店名/菜品/价格。
- **验收**：商户维护店铺+菜品后，做海报时自动填充。

### M3 · 海报主链路（约 1.5 周）
1. 模版海报：模板大厅（行业过滤）+ 对话编辑成品保存为 `PosterWork`。
2. 原生海报：MVP 降级为"更多空白模板任选"。
3. 成品 PNG 导出复用 M0 建好的渲染管线。
- **验收**：两种海报都能生成、保存、导出图片。

### M4 · 营销闭环（约 1 周）
1. 发布管理：作品列表、发布、下载/分享。
2. 统计信息：埋点上报 + 门店统计接口。
- **验收**：商户能发布海报、看到基础数据。

### M5 · PC 后台补齐（约 1 周，可与 M2-M4 并行）
1. 商户管理：门店列表、账号、状态、模板授权。
2. 模板管理：分类、上下架、行业标记、审核流。
- **验收**：运营能建商户、管模板、控制商户可见范围。

> **合计约 6.5 周（单人估算，含联调）**。相比上版计划的 7 周，因去掉了 Taro 前端工作，反而多了 M1 的模板协议扩展 1.5 周。不含微信审核排队时间与独立 Taro 工程周期。

## 10. 待你拍板的决策点

1. **PNG 渲染选型**（M0 硬前置）：Playwright 无头浏览器 vs. SkiaSharp 直接绘制。建议 **Playwright**——复用现有 HTML 渲染视觉 100% 一致、做得快；代价是服务器要装 Chromium、并发吃内存。二期若有量再评估 Skia。
2. **画布规格白名单**：见 §4.3 列出的 8 种规格是否够用？还需要补哪些？（模板本身可以任意尺寸，白名单只影响后台新建模板时的推荐选项）
3. **原生海报**：MVP 就降级为"空白模板任选"可以吗？（否则 M3 多 1 周做 Canvas 自由编辑）
4. **多租户**：一期做多商户 SaaS，还是先单商户跑通？
5. **域名+企业主体**：微信合法域名（HTTPS）是否已备案就绪？（M0 硬前置，未就绪则阻塞小程序端）
6. **Taro 小程序工程排期**：本项目文档只覆盖后端 + PC 后台。**Taro 小程序工程必须并行启动**（一期整体交付的一部分）。谁负责、什么时候启动？建议 M0 结束、API 契约稳定后即可开工。

## 11. 风险与提示

1. **PNG 渲染是硬前置**：现在 `IRenderService` 只输出 HTML，方案 A（后端 PNG）依赖它。M0 就得建好，否则 M2/M3 全卡住。
2. **模板协议扩展与旧模板兼容**：新增 `section` 节点、`background` 对象化后，旧 3 个种子模板（single-dish-red/menu-grid/store-promo）必须仍能渲染——协议演进要向后兼容。
3. **参考图的美术还原度**：客户给的 6 张图美术感很强（背景纹理、艺术字、品牌视觉），系统能做出**结构一致**的模板，但**美术细节**仍需运营人员在 PC 后台配好素材。别承诺"一键还原"。
4. **双模型成本**：文生图按次计费，需加频率限制+缓存。
5. **数据落库**：现有会话部分是内存态，M0 就要补持久化。
6. **PC 后台的位置**：Blazor 后台和 API 共存于同一 ASP.NET Core 应用没问题，但 SignalR 长连接对 Blazor Server 的可靠性要求高。若压力增大，二期考虑 Blazor WASM 或直接换 MVC + Razor 视图。
