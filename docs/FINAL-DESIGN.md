# 商场广告机智能海报系统设计基线

> 适用项目：`O:\test\projects\posters\PosterMint`
> 技术约束：Blazor Server + ASP.NET Core 8 + EF Core + SQLite
> 风格参考：花瓣网（huaban.com）
> 当前定位：商场端制作模板，商铺端仅按权限使用模板

---

## 1. 一句话定义

这是一个面向商场广告机场景的智能海报生产系统：

**商场运营使用 AI 制作和审核模板，商铺只能基于已审核模板通过对话替换内容，最终输出到广告机、H5 和小程序。**

---

## 2. 业务背景

当前商场内海报生产通常依赖第三方设计公司，问题集中在：

1. 成本高，活动越多成本越高。
2. 周期长，无法适应门店高频促销和节日变化。
3. 风格不统一，商场难以做统一品牌治理。

新系统要做的不是“让每个商铺学会设计”，而是：

1. 把海报制作前移到商场端模板生产。
2. 把商铺端工作收敛成内容替换和快速投放。
3. 通过模板、审核、授权、对话替换形成标准化流程。

---

## 3. 需求边界

### 3.1 商场端做什么

商场端负责：

1. 模板分类管理。
2. AI 生成海报背景。
3. AI 生成模板结构和前景元素。
4. 模板打标签。
5. 模板审核。
6. 模板授权给不同商铺或商铺组。

### 3.2 商铺端做什么

商铺端负责：

1. 浏览自己可见的模板。
2. 通过对话替换店名、菜品图、价格、文案、二维码等内容。
3. 让系统自动完成文字长短适配。
4. 导出 H5 / 图片并投放广告机。

### 3.3 商铺端不能做什么

商铺端不能：

1. 创建模板。
2. 修改模板结构。
3. 绕过审核使用未通过模板。
4. 做开放式自由排版设计。

5. 人群标签
6. 人群身份
7. 消费标签

9. 店铺类型
10. 口味标签
11. 营销标签

这套标签在系统里至少驱动四件事：

1. 模板分类和检索
2. 模板推荐
3. 文案生成
4. 风格匹配和投放场景匹配

## 5. 角色与权限

核心角色：

| 角色 | 职责 |
|------|------|
| 超级管理员 | 系统配置、组织管理、模型配置 |
| 商场运营 | 制作模板、编辑模板、提交审核、模板分发 |
| 审核员 | 审核模板、驳回模板、记录审核意见 |
| 商铺管理员 | 选择模板、创建广告内容、导出投放 |
| 商铺编辑员 | 对话替换内容 |
| 商铺查看员 | 只读查看 |

权限原则：

1. 商场端有模板结构权限。
2. 商铺端只有内容替换权限。
3. 模板可见性由审核状态和授权共同决定。

## 7. 前端交互原则

### 7.1 总体视觉方向

界面风格参考花瓣网：

1. 白色背景
2. 高留白
3. 图片驱动
4. 瀑布流 / 卡片流
5. 弱工具感，强内容感

### 7.2 商场后台交互

后台模板制作不应是传统 CMS 表单页，而应是：

1. 左侧实时预览
2. 中间对话式编辑
3. 右侧结构化兜底区
4. 底部版本条

### 7.3 商铺前台交互

商铺端必须以“对话即编辑”为主路径：

1. 默认不暴露字段名和 JSON 结构。
2. 用户像和设计师聊天一样描述需求。
3. 预览实时变化。
4. 表单只能作为异常兜底入口，不能成为主路径。

## 8. AI 分工

### 8.1 为什么要拆成两个模型

本项目必须把文生图模型和文本大模型拆开配置和调用。

原因：

1. 文生图模型适合做背景、氛围、材质感，但不适合做最终文字和二维码。
2. 模板结构和内容替换必须可控、可审计、可回放。
3. 只有文本模型更适合处理布局、字段、版本、审计和适配逻辑。

### 8.2 文生图模型职责

只负责：

1. 商场端背景图生成
2. 背景风格变化
3. 背景中的留白区氛围控制

不负责：

1. 最终文字
2. 最终价格
3. 二维码
4. 模板结构落库

### 8.3 文本模型职责

负责：

1. 商场端根据对话生成模板结构
2. 调整前景元素位置和样式
3. 商铺端内容替换
4. 文字长短适配
5. 文案生成
6. 自动预审

### 8.4 推荐的智能体

| 智能体 | 职责 |
|------|------|
| `BackgroundAgent` | 背景生成 prompt 与背景任务 |
| `LayoutAgent` | 商场端模板结构生成与调整 |
| `ContentAgent` | 商铺端内容替换和样式适配 |
| `CopyAgent` | 文案生成 |
| `ReviewAgent` | 自动预审 |
| `RecommendAgent` | 标签推荐模板 |

## 9. 模板 JSON 协议

### 9.1 必须保留的层级

新版必须保留现有项目的三层结构：

```json
{
  "canvas": {},
  "fields": [],
  "layout": []
}
```

原因：

1. 渲染可控
2. 模板和内容天然分离
3. 商铺替换内容时有明确边界
4. 后续对布局和字段进行精细调整时不需要重构协议

### 9.2 新版模板建议补充的字段

在不破坏现有层级的前提下补充：

1. `categoryId`
2. `thumbnail`
3. `tags`
4. `aiHints`
5. `meta`

### 9.3 字段层语义

`fields` 只描述商铺可以替换的内容，如：

1. 店名
2. 标题 / 副标题 / slogan
3. 菜品图 / 店铺图
4. 原价 / 活动价
5. 地址 / 电话
6. 二维码
7. 卖点数组
8. 菜单数组

### 9.4 布局节点白名单

推荐继续使用白名单节点：

1. `rect`
2. `text`
3. `price`
4. `image`
5. `qrcode`
6. `repeat`

## 11. 核心流程

### 11.1 商场后台模板生产

```text
选分类 / 选场景
  -> 对话描述海报目标
  -> 文生图生成背景
  -> 文本模型生成前景结构
  -> 实时预览
  -> 打标签
  -> 保存草稿
  -> 自动预审
  -> 人工审核
  -> 审核通过
  -> 模板分发给商铺
```

### 11.2 商铺前台内容替换

```text
登录
  -> 看到自己可见模板
  -> 选择模板
  -> 创建会话
  -> 对话修改内容
  -> AI 自动适配长短文字和局部样式
  -> 实时预览
  -> 版本回看
  -> 导出 H5 / 图片
```

### 11.3 审核流

```text
Draft -> Pending -> Approved
              \-> Rejected -> Draft
```

### 11.4 会话流

```text
Created -> Editing -> Exported
              \-> Archived
```

## 13. 渲染与输出

### 13.1 输出格式

必须稳定支持：

1. HTML
2. SVG
3. PNG / JPG

### 13.2 输出用途

| 格式 | 用途 |
|------|------|
| HTML | H5、广告机浏览器、小程序 `web-view` |
| SVG | 高清矢量导出 |
| PNG / JPG | 广告机图片、分享图、静态物料 |

### 13.3 广告机要求

1. 1080x1920 为优先画布
2. H5 必须可自适应 WebView
3. 图片导出支持 2x / 3x
4. 预览和导出视觉要尽量一致

## 14. 部署要求

项目必须满足：

1. Blazor 开发
2. SQLite 数据库
3. 样式参考花瓣网
4. 支持文本模型与图像模型独立配置

```text

```

## 15. 对当前新项目的直接约束

`PosterMint` 实现时必须遵守：

1. 商铺端默认入口应该是场景 / 对话入口，不是模板字段表单。
2. 模板广场是后备路径，不应成为主心智。
3. 会话里必须保存消息、版本和建议动作，而不是一次性字符串替换。
4. 后台必须补出真正的模板生产页、审核页和授权页。
5. 当前已做的本地对话引擎只是演示底座，后续要接入正式外部模型。

---

## 16. 当前设计结论

这个项目的正确方向不是“做一个通用 AI 海报工具”，而是：

1. 商场端把 AI 变成模板生产力工具。
2. 商铺端把 AI 变成模板内容替换助手。
3. 模板、权限、审核、标签、导出共同构成广告生产闭环。

最终系统必须让非专业运营和门店人员都能完成工作，同时保证商场对内容质量和品牌秩序有控制权。
